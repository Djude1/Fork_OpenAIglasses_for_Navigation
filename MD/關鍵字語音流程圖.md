# 關鍵字語音觸發流程

> 本文件說明 AI 智慧眼鏡系統中，語音關鍵字（「曼波」）的完整觸發流程。
> 包含 ESP32 眼鏡模式與 APP 旁路模式兩種路徑。

---

## 一、ESP32 眼鏡模式（語音喚醒）

### 1.1 收音流程

```
使用者說話 → ESP32 麥克風收音
    → WebSocket /ws_audio → send_audio_frame()
    → AGC 自適應增益（音量太小放大、太大壓縮）
    → 待機靜音門檻（RMS < 80 丟棄，不送 ASR）
    → 放進 audio_queue
    → Google STT 串流辨識（cmn-Hant-TW，熱詞 boost=20）
    → _handle_result() 判斷待機/主動模式
```

### 1.2 待機模式（standby）下的觸發流程

```
Google STT 回傳 final 結果
  → _check_wake_word(text)
    → _normalize_cn(text)     ← 簡→繁轉換 + 去標點
    → 檢查是否包含 _MAMBO_VARIANTS 中的任一變體：
       ("曼波", "漫波", "漫播", "慢播", "慢波", "滿波", "們波")
    → 有匹配？
       ├─ 是 → 說話人驗證（若啟用）→ on_wake() + enter_active_mode()
       └─ 否 → 印 log「無喚醒詞，忽略」
```

### 1.3 主動模式（active）下的流程

```
Google STT 回傳結果（partial 和 final 都處理）
  → _handle_active(text, is_final)
    → final 結果 → 檢查 END_WORDS（「謝謝曼波」等）
       ├─ 匹配 → on_end_word() → 切回 standby
       └─ 不匹配 → 當作指令派發給 AI
    → 靜音超過 SILENCE_SEC 或超時 → 自動切回 standby
```

---

## 二、APP 旁路模式（手機直連）

```
APP 麥克風收音 → WebSocket /ws_audio
  → bypass_wake = true（APP 連線時自動啟用）
  → _check_wake_word() 直接跳過喚醒詞判斷
  → STT 結果直接派發給 AI（on_event）
  → 使用者不需要說「曼波」，直接說指令即可
```

## 四、Google STT 熱詞加成

`speech_contexts` 設定 `boost=20.0`，讓 Google STT 優先把相似音辨識為以下詞彙：

```
曼波、漫波、漫播、慢播、慢波、滿波、們波
哈囉曼波、哈囉 曼波、哈羅曼波、哈洛曼波
哈囉慢波、哈囉漫波、哈囉滿波、哈囉慢播
謝謝曼波、謝謝 曼波、羅曼波、哈囉嗎
```

這些熱詞的目的是**降低 ASR 把「曼波」誤辨為「半波」「門波」等不相關詞的機率**，不是匹配用的。
實際的觸發判斷完全由 `_MAMBO_VARIANTS` 控制。

## 七、設計決策記錄

### 為什麼關鍵字是「曼波」而不是「哈囉曼波」？

使用者明確指示：「主要的關鍵字是曼波，不是哈囉漫波，所以監測到漫波就要觸發」。

這意味著：
- **「哈囉」不是必要的** — 使用者可以直接說「曼波」來喚醒，不必加「哈囉」前綴
- **任何含「曼波」的文字都會觸發** — 包含「曼波舞」「跳曼波」等日常用語
- **「哈囉」單獨不會觸發** — 說「哈囉世界」「哈囉你好」不會啟動 AI

### 為什麼「謝謝曼波」在待機模式下也會觸發？

因為「謝謝曼波」包含「曼波」，所以在待機模式下會觸發喚醒。
在主動模式下，「謝謝曼波」會被 END_WORDS 優先攔截，觸發結束對話。
這是設計上可接受的行為 — 使用者不太可能在待機時無緣無故說「謝謝曼波」。

### 為什麼需要 7 個變體？

Google STT 的 ASR 可能將「曼波」誤辨為：
- **漫波** — 曼→漫（最常見）
- **漫播** — 曼→漫，波→播
- **慢播** — 曼→慢，波→播
- **慢波** — 曼→慢
- **滿波** — 曼→滿
- **們波** — 曼→們

這些都是實測中觀察到的 ASR 誤辨結果，必須全部涵蓋才能保證喚醒成功率。
